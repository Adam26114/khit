1. Project Overview

1.1 Executive Summary
This document defines the full product requirements for a Myanmar local brand shirt e-commerce platform. The platform draws design and UX inspiration from Mango (shop.mango.com), adapting its clean, minimal aesthetic for a local Myanmar fashion brand. The platform will serve customers browsing and purchasing shirts online, while providing administrators with full control over inventory, orders, and storefront content.

1.2 Project Goals
	•	Build a world-class, minimal white e-commerce experience for a Myanmar local shirt brand
	•	Deliver a fully functional storefront with product browsing, filtering, and checkout
	•	Provide robust admin controls for product, order, inventory, and user management
	•	Support Myanmar-specific delivery options: nationwide shipping and store pickup
	•	Accept Cash on Delivery (COD) as the sole payment method at launch
	•	Establish error monitoring, code quality gates, and authentication best practices

2.1 Design System Principles
	•	Package manager: Bun — use bun install, bun run, bun add, bun x (never npm or yarn)
	•	Color palette: White (#FFFFFF) primary background, Black (#000000) primary text, Light Gray (#F5F5F5) surfaces
	•	Typography: Neue Helvetica Paneuropean — Regular for body, Medium for labels, Bold for headings
	•	Icons: Phosphor Duotone — consistent weight and two-tone treatment across all components
	•	Spacing: 4px base unit, multiples of 4 throughout (8, 12, 16, 24, 32, 48, 64, 96)
	•	Border radius: 2px for product cards, 4px for buttons, 8px for modals
	•	Motion: Minimal — 150ms ease transitions only; no decorative animations

Key Relationships & Business Rules
	•	Each product variant belongs to ONE product and ONE color
	•	Each product variant can have MANY media items (images/videos)
	•	Media carousel shows all images for selected variant on hover
	•	Color change = variant change = image set change
	•	Size buttons show availability from variant stock_quantity
	•	First color in variant set is default display color
	•	Primary image (is_primary=true) shown as product card thumbnail

7.3 Error Handling Patterns
Frontend (Next.js)
	•	Use Next.js error.tsx boundaries at route segment level
	•	Global error.tsx at app root catches unhandled rendering errors
	•	All Convex query/mutation calls wrapped in try-catch with user-friendly toast notifications
	•	Form validation uses Zod schemas; errors displayed inline below fields
	•	Network errors display retry UI — never silent failures

Backend (Convex)
	•	All mutations validate input with Convex validators — reject invalid data at API boundary
	•	Business logic errors return structured error objects with error codes
	•	Stock deduction uses optimistic concurrency — race conditions handled gracefully
	•	Order creation is atomic — partial failures roll back completely

User-Facing Error States
Scenario
User Experience
Product out of stock (race condition at checkout)
Toast: 'Item sold out. Removed from cart.' Cart updates.
Network offline
Persistent banner: 'No internet connection. Changes will sync when back online.'
Checkout form validation
Inline red error text below each invalid field; form does not submit
Server error on order submit
Toast: 'Unable to place order. Please try again.' Order not created.
Admin: failed status update
Toast: 'Failed to update. Refreshing order data...' — revert optimistic update

10.2 Pull Request Process
	•	Every PR requires CodeRabbit review to pass before merge
	•	Minimum 1 human reviewer approval required
	•	PRs must reference a GitHub Issue number in the title (e.g., feat: #42 — add product filter)
	•	All PRs target develop; only release PRs target main
	•	Sentry release tracking: deploy hook fires on merge to main, creating a new Sentry release
	•	CI runs: bun install --frozen-lockfile && bun run build — PRs blocked if either fails
	•	Lockfile: bun.lockb must always be committed; PRs without it are rejected
